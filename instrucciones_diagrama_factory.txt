INSTRUCCIONES PARA GENERAR EL DIAGRAMA UML DEL PATRÓN FACTORY
=============================================================

[Formato sugerido: Documento de instrucciones para IA generadora de imágenes]
Este documento describe con detalle cómo generar una imagen tipo draw.io para el diagrama UML del patrón Factory / Fábrica Simple aplicado al sistema EducationMaxxingMVP. La estética debe imitar el estilo visual del diagrama de arquitectura usado como referencia: fondo negro, cajas con bordes blancos, texto blanco, líneas rectas, flechas claras, separación por secciones y leyenda visual.


OBJETIVO DE LA IMAGEN
=====================

[Formato sugerido: Párrafo]
Crear un diagrama UML de clases que muestre cómo UserService.CreateUser funciona como una Fábrica Simple para crear usuarios concretos del dominio: Student, Professor y Admin. El diagrama debe verse como si hubiera sido creado manualmente en draw.io, con una estética técnica, ordenada, monocromática y de alto contraste. Debe representar clases reales del sistema, atributos, métodos, relaciones y una pequeña leyenda de líneas.


ESTILO VISUAL GENERAL
=====================

[Formato sugerido: Lista de viñetas]
- Usar fondo negro o gris muy oscuro, casi negro.
- Usar texto blanco.
- Usar cajas rectangulares con borde blanco fino.
- Usar esquinas ligeramente redondeadas, pero no demasiado redondeadas.
- Usar líneas blancas para relaciones.
- Usar flechas sólidas para llamadas directas o creación.
- Usar flechas punteadas para dependencia o implementación de interfaz.
- Usar flechas de generalización UML con triángulo hueco para herencia/composición hacia User.
- Mantener estilo limpio, técnico y similar a un diagrama de draw.io.
- No usar colores brillantes ni degradados.
- No usar ilustraciones decorativas.
- No usar iconos innecesarios.
- Mantener alineación estricta en columnas y filas.
- La imagen debe tener aspecto de diagrama académico de ingeniería de software.


FORMATO DE LIENZO
=================

[Formato sugerido: Lista de especificaciones]
- Relación de aspecto recomendada: 16:9 horizontal.
- Resolución sugerida: 1400 x 900 px o similar.
- Margen exterior: 30 px.
- Fondo: #111111.
- Bordes: blanco #FFFFFF, grosor 1.2 px.
- Texto: blanco #FFFFFF.
- Fuente: sans-serif técnica similar a Arial, Helvetica o Roboto.
- Tamaño de texto para títulos: 17 a 20 px.
- Tamaño de texto para nombres de clases: 15 a 17 px.
- Tamaño de texto para atributos y métodos: 12 a 14 px.
- Alinear texto dentro de las cajas a la izquierda, excepto títulos generales y encabezados de sección, que deben ir centrados.


DISTRIBUCIÓN GENERAL DEL DIAGRAMA
=================================

[Formato sugerido: Instrucciones de composición]
Dividir el diagrama en tres zonas horizontales principales y una zona lateral de leyenda.

Zona 1: Título superior
-----------------------
- En la parte superior central colocar el título:
  "Factory / Fábrica Simple - Creación de usuarios"
- Debajo del título, en texto más pequeño, colocar:
  "EducationMaxxingMVP - src/app/user y src/domain/user"
- El título debe estar dentro de una caja delgada o directamente sobre el fondo, pero centrado.

Zona 2: Fábrica y entrada
-------------------------
- En la parte superior izquierda colocar la clase CreateUserInputDTO.
- En la parte superior central colocar la clase UserService con una etiqueta visible:
  "<<Factory>>"
- UserService debe ser visualmente la caja más importante de esta zona.
- Dibujar una flecha sólida desde CreateUserInputDTO hacia UserService.
- Etiqueta de la flecha:
  "recibe input"

Zona 3: Productos de dominio
----------------------------
- En el centro del diagrama colocar la clase User como producto base.
- Debajo de User, colocar tres cajas en fila:
  Student, Professor y Admin.
- Student debe estar abajo a la izquierda.
- Professor debe estar abajo al centro.
- Admin debe estar abajo a la derecha.
- Dibujar relaciones UML desde Student, Professor y Admin hacia User.
- Las flechas deben apuntar hacia User.
- Etiqueta general cerca de estas flechas:
  "hereda / embebe User"

Zona 4: Persistencia e interfaz
-------------------------------
- A la derecha de UserService colocar la interfaz UserRepository.
- Debajo o a la derecha de UserRepository colocar la clase UserRecord.
- Dibujar una flecha punteada desde UserService hacia UserRepository.
- Etiqueta:
  "depende de interfaz"
- Dibujar una flecha sólida desde UserService hacia UserRecord.
- Etiqueta:
  "crea registro persistible"
- Dibujar una flecha punteada desde UserRecord hacia UserRepository o una flecha sólida desde UserService hacia UserRepository con etiqueta:
  "Save(record)"

Zona 5: Leyenda
---------------
- En la parte inferior derecha colocar una caja titulada:
  "Leyenda de líneas"
- Dentro de la leyenda incluir tres ejemplos:
  1. Línea sólida con flecha: "crea / llama"
  2. Línea punteada con flecha: "depende de"
  3. Línea con triángulo hueco UML: "hereda / embebe"

ORGANIZACIÓN ESPACIAL RECOMENDADA
=================================

[Formato sugerido: Descripción por coordenadas aproximadas]
Usar una organización parecida a esta:

- Título en la parte superior central.
- CreateUserInputDTO en columna izquierda superior.
- UserService al centro superior.
- UserRepository en columna derecha superior.
- User en el centro medio.
- UserRecord a la derecha media, debajo de UserRepository.
- Student, Professor y Admin en una fila inferior.
- Leyenda en la esquina inferior derecha.

[Formato sugerido: Diagrama textual de referencia]

                 [Título: Factory / Fábrica Simple - Creación de usuarios]

 [CreateUserInputDTO] ---> [<<Factory>> UserService] - - - -> [<<interface>> UserRepository]
                                  |       |       |                  ^
                                  |       |       |                  |
                                  |       |       +-----> [UserRecord]
                                  |       |
                                  v       v
                              [Student] [Professor] [Admin]
                                   \       |       /
                                    \      |      /
                                     \     |     /
                                      v    v    v
                                      [User]

                                                [Leyenda de líneas]

Nota: En la imagen final puede invertirse la posición de User y los productos concretos para lograr una lectura UML más tradicional, siempre que las flechas de herencia apunten hacia User.

RESTRICCIONES IMPORTANTES
=========================

[Formato sugerido: Lista de viñetas]
- No usar clases genéricas como Creator, Product, ConcreteProduct o FactoryProduct.
- Usar únicamente nombres reales del sistema.
- No representar Singleton en este diagrama.
- No agregar base de datos, HTTP handlers ni capas completas de arquitectura; este diagrama es solo del patrón Factory.
- No usar colores de relleno fuertes.
- No usar sombras pesadas.
- No usar dibujos decorativos.
- No saturar el diagrama con demasiado texto pequeño.
- Mantener el aspecto similar a draw.io, no a una infografía publicitaria.
- La imagen debe parecer un diagrama técnico formal para entregar en clase.
